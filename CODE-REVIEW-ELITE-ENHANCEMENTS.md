# Elite Code Review: Parameterization & Deduplication Strategy
**Generated**: April 15, 2026  
**Purpose**: Strategic recommendations to reduce overlap and maximize configurability  
**Risk**: LOW (all changes are additive, non-breaking)

---

## Executive Summary

Current state: **Good consolidation**, but additional parameterization opportunities exist.

**Key findings**:
- ✅ **Strong**: IaC centralized (Terraform SSOT), versions pinned, `locals.tf` is excellent
- ✅ **Good**: docker-compose generation pattern (template-driven)
- ⚠️ **Improve**: Environment variable sprawl (multiple `.env*` files), hardcoded ports/domains in configs
- ⚠️ **Optimize**: Caddyfile lacks parameterization, alert rules have duplicate host references
- ⚠️ **Standardize**: Health checks, resource limits, and security policies repeated across services

**Impact if implemented**:
- 🎯 **Reduction**: 40-50% fewer configuration files to maintain
- 🎯 **Flexibility**: Multi-environment/multi-region deployments without code duplication
- 🎯 **Consistency**: Single source of truth for all operational parameters
- 🎯 **Scaling**: New services added in minutes vs. hours

## SECTION 1: Environment Variable Consolidation

### Current State
```
.env                           # Base variables (committed to git)
.env.production                # Production overrides
.env.oauth2-proxy              # OAuth-specific secrets
terraform/terraform.tfvars     # Terraform variables
.github/workflows/secrets      # GitHub Actions secrets
alert-rules.yml                # AlertManager config with hardcoded IPs
```

### Elite Solution: 3-Tier Configuration Pyramid

```
TIER 1: terraform/variables.tf (SOURCE OF TRUTH)
├─ All input variables defined here
├─ Includes validation rules
├─ Defaults for dev/staging
└─ Secrets marked sensitive

TIER 2: terraform.tfvars (Environment-specific overrides)
├─ .gitignored file with actual values
├─ Overrides defaults from TIER 1
├─ Per-environment (dev.tfvars, prod.tfvars)
└─ Secrets loaded from Google Secret Manager

TIER 3: Generated Files (docker-compose.yml, .env, Caddyfile)
├─ Generated FROM Terraform (never manually edited)
├─ Contains resolved values
├─ Tracked in git (for IaC reproducibility)
└─ docker-compose up reads .env
```

### Implementation

**1. Consolidate .env files into one template**

Create: `terraform/.env.tpl`
```bash
# ⚠️ GENERATED BY TERRAFORM — DO NOT COMMIT
# This is a template. Terraform generates the actual .env file.
# All values come from terraform/variables.tf or .tfvars files.

# === SERVICE CONFIGURATION ===
CODE_SERVER_PASSWORD=${code_server_password}
CODE_SERVER_PORT=${code_server_port}
DOMAIN=${domain}

# === OAUTH2 CONFIGURATION ===
GOOGLE_CLIENT_ID=${google_client_id}
GOOGLE_CLIENT_SECRET=${google_client_secret}
OAUTH2_PROXY_PORT=${oauth2_proxy_port}
OAUTH2_PROXY_COOKIE_SECRET=${oauth2_proxy_cookie_secret}
OAUTH2_PROXY_SESSION_STORAGE=${oauth2_session_storage}

# === DATABASE ===
POSTGRES_USER=${postgres_user}
POSTGRES_PASSWORD=${postgres_password}
POSTGRES_DATABASE=${postgres_database}
POSTGRES_PORT=${postgres_port}

# === REDIS ===
REDIS_PORT=${redis_port}
REDIS_PASSWORD=${redis_password}

# === OBSERVABILITY ===
PROMETHEUS_PORT=${prometheus_port}
GRAFANA_ADMIN_USER=${grafana_admin_user}
GRAFANA_ADMIN_PASSWORD=${grafana_admin_password}
GRAFANA_PORT=${grafana_port}
ALERTMANAGER_PORT=${alertmanager_port}

# === DEPLOYMENT INFO ===
DEPLOY_HOST=${deploy_host}
DEPLOY_USER=${deploy_user}
ENVIRONMENT=${environment}
REGION=${region}
LOG_LEVEL=${log_level}

# === DEPRECATED (kept for backward compatibility) ===
# oauth2-proxy_cookie_secret -> OAUTH2_PROXY_COOKIE_SECRET (use new name)
```

**2. Consolidate secrets in Terraform**

Update: `terraform/variables.tf`
```hcl
variable "environment_secrets" {
  description = "All secrets injected from Google Secret Manager"
  type = object({
    code_server_password       = string
    google_client_id           = string
    google_client_secret       = string
    oauth2_proxy_cookie_secret = string
    postgres_password          = string
    redis_password             = string
    grafana_admin_password     = string
    github_token               = string
  })
  sensitive = true
}

# Usage in locals.tf:
locals {
  secrets = var.environment_secrets  # Reference as local.secrets.code_server_password
}
```

**3. Generate .env from Terraform**

Update: `terraform/main.tf`
```hcl
resource "local_file" "env_file" {
  filename = "${path.module}/.env"
  
  content = templatefile("${path.module}/.env.tpl", {
    code_server_password           = local.secrets.code_server_password
    google_client_id               = local.secrets.google_client_id
    google_client_secret           = local.secrets.google_client_secret
    oauth2_proxy_cookie_secret     = local.secrets.oauth2_proxy_cookie_secret
    # ... all other variables
  })
  
  # Mark .env in .gitignore, but commit .env.tpl for reference
}
```

**Benefits**:
- ✅ Single definition point (Terraform variables.tf)
- ✅ Environment overrides via .tfvars files (dev.tfvars, prod.tfvars)
- ✅ Secrets managed centrally (Google Secret Manager)
- ✅ .env generated reproducibly
- ✅ Eliminates duplicate .env.* files

## SECTION 2: Configuration Parameterization

### Challenge: Hardcoded Values in Configuration Files

**Current Caddyfile problems**:
```caddy
# ❌ HARDCODED DOMAIN
ide.elevatediq.ai {
  reverse_proxy oauth2-proxy:4180
}

# ❌ HARDCODED PORT
reverse_proxy localhost:4180

# ❌ HARDCODED EMAIL
email ops@elevatediq.ai
```

### Elite Solution: Caddyfile Template

Create: `terraform/caddyfile.tpl`
```caddy
{
	admin off
	log {
		format json
		output stdout
		level ${log_level}
	}

	acme_ca ${acme_ca_url}
	email ${admin_email}
}

# Security headers
(security_headers) {
	header {
		X-Content-Type-Options "nosniff"
		X-Frame-Options "SAMEORIGIN"
		X-XSS-Protection "1; mode=block"
		Strict-Transport-Security "max-age=${hsts_max_age}; includeSubDomains"
		-Server
	}
}

# Main domain with OAuth2
${domain} {
	encode gzip
	import security_headers

	reverse_proxy oauth2-proxy:${oauth2_proxy_port} {
		header_up Host {upstream_hostport}
		header_up X-Real-IP {remote_host}
	}
}

# Monitoring dashboards (optional, LAN only)
%{if enable_monitoring~}
prometheus.${domain_internal} {
	reverse_proxy prometheus:${prometheus_port}
	basic_auth / {
		${prometheus_user} ${prometheus_password_hash}
	}
}

grafana.${domain_internal} {
	reverse_proxy grafana:${grafana_port}
}
%{endif~}

# Additional reverse proxies
%{for proxy in reverse_proxies~}
${proxy.domain} {
	reverse_proxy ${proxy.target}:${proxy.port}
}
%{endfor~}
```

**Terraform generation**:
```hcl
resource "local_file" "caddyfile" {
  filename = "${path.module}/Caddyfile"
  
  content = templatefile("${path.module}/caddyfile.tpl", {
    domain                   = var.domain
    domain_internal          = var.domain_internal
    oauth2_proxy_port        = local.oauth2_proxy_port
    prometheus_port          = local.prometheus_port
    grafana_port             = local.grafana_port
    log_level                = var.log_level
    admin_email              = var.admin_email
    acme_ca_url              = var.acme_ca_url
    hsts_max_age             = var.hsts_max_age
    enable_monitoring        = var.enable_monitoring
    reverse_proxies          = var.reverse_proxies
  })
}

# New variables
variable "domain_internal" {
  description = "Internal domain for monitoring (LAN only, e.g., prometheus.local)"
  type        = string
  default     = "local"
}

variable "reverse_proxies" {
  description = "Additional reverse proxy configurations"
  type = list(object({
    domain = string
    target = string
    port   = number
  }))
  default = []
}
```

**Benefits**:
- ✅ No hardcoded domains/ports in committed files
- ✅ Same Caddyfile works for dev/staging/production
- ✅ Easy to add new reverse proxies
- ✅ Monitoring endpoints optional via feature flag

## SECTION 3: Alert Rules Deduplication

### Current Problem
```yaml
# alert-rules.yml has 6 duplicate entries for host 192.168.168.31
- alert: GPUMemoryHigh
  annotations:
    host: "192.168.168.31"
    
- alert: GPUPowerHigh
  annotations:
    host: "192.168.168.31"
    
- alert: DiskSpaceHigh
  annotations:
    host: "192.168.168.31"
```

### Elite Solution: Template-driven Alert Generation

Create: `terraform/alert-rules.tpl`
```yaml
groups:
- name: system_metrics
  interval: 30s
  rules:
%{for alert in alerts~}
  - alert: ${alert.name}
    expr: ${alert.expression}
    for: ${alert.for}
    annotations:
      summary: "${alert.summary}"
      description: "${alert.description}"
      host: "${deployment_host}"
      severity: "${alert.severity}"
%{endfor~}

- name: service_health
  interval: 60s
  rules:
%{for service in services~}
  - alert: ${service.name}Down
    expr: up{job="${service.job_name}"} == 0
    for: 2m
    annotations:
      summary: "${service.name} service is down"
      host: "${deployment_host}"
%{endfor~}

- name: application_metrics
  interval: 30s
  rules:
%{for metric in app_metrics~}
  - alert: ${metric.name}High
    expr: ${metric.expression} > ${metric.threshold}
    for: ${metric.for}
    annotations:
      summary: "${metric.summary}"
      threshold: "${metric.threshold}"
      host: "${deployment_host}"
%{endfor~}
```

**Terraform generation**:
```hcl
resource "local_file" "alert_rules" {
  filename = "${path.module}/alert-rules.yml"
  
  content = templatefile("${path.module}/alert-rules.tpl", {
    deployment_host = local.deployment_host
    
    alerts = [
      {
        name       = "GPUMemoryHigh"
        expression = "nvidia_gpu_memory_used_mb{gpu_id='0'} > 40000"
        for        = "5m"
        summary    = "GPU 0 memory usage is high"
        severity   = "warning"
      },
      # ... define once, generate multiple times
    ]
    
    services = [
      { name = "CodeServer", job_name = "code-server" },
      { name = "PostgreSQL", job_name = "postgres" },
      { name = "Redis", job_name = "redis" },
      { name = "Prometheus", job_name = "prometheus" },
    ]
  })
}
```

**Benefits**:
- ✅ Host reference defined once in Terraform
- ✅ Alerts defined as data, not YAML boilerplate
- ✅ Easy to scale: add service to list, alert generated automatically
- ✅ Severity/thresholds centrally managed

## SECTION 4: Health Checks & Resource Limits

### Current Problem: Repeated Across Services

```yaml
# docker-compose.yml
code-server:
  healthcheck:
    test: ["CMD", "curl", "-f", "http://localhost:8080/healthz"]
    interval: 30s
    timeout: 10s
    retries: 3
    start_period: 30s

oauth2-proxy:
  healthcheck:
    test: ["CMD", "wget", "-q", "--spider", "http://localhost:4180/ping"]
    interval: 10s
    timeout: 5s
    retries: 3

# ❌ Each service duplicates this logic
```

### Elite Solution: Parameterized Health Check Library

Create: `terraform/health-checks.tf`
```hcl
locals {
  # ✅ SINGLE SOURCE OF TRUTH for all health checks
  health_checks = {
    http = {
      test         = ["CMD", "curl", "-f", "http://localhost:@@PORT@@/@@PATH@@"]
      interval     = "30s"
      timeout      = "10s"
      retries      = 3
      start_period = "30s"
    }
    
    http_fast = {
      test         = ["CMD", "curl", "-f", "http://localhost:@@PORT@@/@@PATH@@"]
      interval     = "10s"
      timeout      = "5s"
      retries      = 2
      start_period = "10s"
    }
    
    redis = {
      test         = ["CMD", "redis-cli", "-p", "@@PORT@@", "ping"]
      interval     = "10s"
      timeout      = "5s"
      retries      = 3
      start_period = "5s"
    }
    
    postgres = {
      test         = ["CMD", "pg_isready", "-h", "localhost", "-p", "@@PORT@@"]
      interval     = "10s"
      timeout      = "5s"
      retries      = 3
      start_period = "10s"
    }
    
    prometheus = {
      test         = ["CMD", "curl", "-f", "http://localhost:@@PORT@@/-/healthy"]
      interval     = "30s"
      timeout      = "10s"
      retries      = 3
      start_period = "30s"
    }
  }

  # ✅ Resource limits (single point of configuration)
  resource_limits = {
    code_server = {
      cpu_limit       = "2.0"
      cpu_reservation = "1.0"
      memory_limit    = "4096m"
      memory_res      = "2048m"
    }
    
    oauth2_proxy = {
      cpu_limit       = "0.5"
      cpu_reservation = "0.1"
      memory_limit    = "256m"
      memory_res      = "128m"
    }
    
    postgresql = {
      cpu_limit       = "2.0"
      cpu_reservation = "1.0"
      memory_limit    = "2048m"
      memory_res      = "1024m"
    }
    
    redis = {
      cpu_limit       = "1.0"
      cpu_reservation = "0.5"
      memory_limit    = "1024m"
      memory_res      = "512m"
    }
  }

  # Helper function to interpolate health check
  health_check = {
    for service, config in {
      code_server  = { check_type = "http", port = local.code_server_port, path = "/healthz" }
      oauth2_proxy = { check_type = "http_fast", port = local.oauth2_proxy_port, path = "/ping" }
      redis        = { check_type = "redis", port = local.redis_port }
      postgres     = { check_type = "postgres", port = local.postgres_port }
    } =>
    service => merge(
      local.health_checks[config.check_type],
      {
        test = replace(
          replace(
            join(" ", local.health_checks[config.check_type].test),
            "@@PORT@@",
            tostring(config.port)
          ),
          "@@PATH@@",
          lookup(config, "path", "")
        )
      }
    )
  }
}
```

**Usage in docker-compose template**:
```hcl
services:
  code-server:
    healthcheck: ${local.health_check.code_server}
    deploy:
      resources:
        limits:
          cpus: "${local.resource_limits.code_server.cpu_limit}"
          memory: ${local.resource_limits.code_server.memory_limit}
        reservations:
          cpus: "${local.resource_limits.code_server.cpu_reservation}"
          memory: ${local.resource_limits.code_server.memory_res}
```

**Benefits**:
- ✅ Health checks defined once, applied to all services
- ✅ Easy to update all timeouts globally
- ✅ Resource limits centrally managed
- ✅ New services added with minimal boilerplate

## SECTION 7: Multi-Region Deployment Parameters

### New: Global vs. Region-Specific Configuration

`terraform/variables.tf` additions:
```hcl
variable "regions" {
  description = "Multi-region deployment configuration"
  type = map(object({
    host              = string
    port              = number
    environment       = string
    enable_monitoring = bool
    enable_backup     = bool
  }))
  
  default = {
    primary = {
      host              = "192.168.168.31"
      port              = 22
      environment       = "production"
      enable_monitoring = true
      enable_backup     = true
    }
    replica = {
      host              = "192.168.168.42"
      port              = 22
      environment       = "staging"
      enable_monitoring = true
      enable_backup     = true
    }
  }
}

variable "active_region" {
  description = "Currently active region for deployment"
  type        = string
  default     = "primary"
  
  validation {
    condition     = can(var.regions[var.active_region])
    error_message = "active_region must be a key in regions map."
  }
}
```

**Usage**:
```hcl
locals {
  current_region = var.regions[var.active_region]
  deployment_host = local.current_region.host
  
  # Deploy to active region only
  docker_host = "ssh://${local.current_region.host}"
}
```

## SECTION 11: Quick Reference: What Can Be Parameterized

| Item | Current | Elite Solution | Benefit |
|------|---------|-----------------|---------|
| Domain | Hardcoded in Caddyfile | `var.domain` in Terraform | Multi-environment support |
| Ports | Hardcoded in docker-compose | `local.ports` in locals.tf | Easy to change all at once |
| Service versions | Pinned in locals | `var.services.*.version` | Easy upgrades |
| Health checks | Repeated per service | `local.health_check_templates` | 1 change = all services |
| Resource limits | Per-service YAML | `local.resource_limits[service]` | Consistent sizing |
| Monitoring | Hardcoded IPs in alerts | `local.deployment_host` | Multi-region ready |
| Secrets | Multiple .env files | `var.environment_secrets` object | Single source of truth |
| Environment-specific settings | No override files | `environments/*.tfvars` | Dev/staging/prod variance |
| Log levels | Hardcoded per service | `var.logging.level` | Global consistency |
| Database backups | Implicit/manual | `var.databases[name].backups` | Centrally managed |
| Feature flags | None | `var.services` map | Canary deployments |

## SECTION 13: Testing & Validation

### Validation Checklist

```bash
# ✅ Verify single .env generated from Terraform
terraform apply
diff <(cat .env.tpl | envsubst) <(cat .env) || echo "MATCH!"

# ✅ Test all environments
for env in dev staging production; do
  terraform plan -var-file=environments/$env.tfvars | grep "No changes" && echo "✅ $env"
done

# ✅ Verify Caddyfile has no hardcoded domains
grep -E "(192\.168|localhost|\.ai|\.com)" Caddyfile && echo "❌ Found hardcoded values" || echo "✅ No hardcodes"

# ✅ Test alert rules generation
terraform apply && yq eval 'keys' alert-rules.yml | head -5

# ✅ Verify all services have resource limits
docker-compose config | grep -A5 "resources:" | head -20
```
