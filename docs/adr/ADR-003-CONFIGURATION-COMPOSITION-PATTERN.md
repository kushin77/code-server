# ADR-003: Configuration Composition Pattern (Terraform → Docker Compose)

**Status**: Accepted  
**Deciders**: @kushin77  
**Context**: Phase 2 Code Consolidation — achieving IaC compliance for all services

---

## Context and Problem Statement

The repository previously had 10+ manually managed Docker Compose files spread across:
- Phase-specific files (`docker-compose-phase-15.yml`, `docker-compose-phase-16-deploy.yml`)
- Variant-specific files (base/production/debug)
- Ad-hoc monitoring service starts (prometheus, grafana, postgres, redis running outside IaC)

This created:
- **Drift**: Running services not matching committed configuration
- **Duplication**: 40%+ repeated service definitions
- **Immutability violation**: `latest` tags and manually started containers
- **Operational risk**: No single source of truth for what is deployed

---

## Decision Drivers

- IaC: all infrastructure must be version-controlled and reproducible
- Immutability: exact version pins prevent silent upgrades
- Idempotency: `terraform apply` must be safe to re-run at any time
- No duplication: one definition per service, zero copy-paste
- On-prem focus: no external dependencies (cloud APIs, registries) required for deployment

---

## Considered Options

### Option A: Multiple static docker-compose.yml files per environment
- ❌ Duplication inevitable
- ❌ Versions drift between files
- ❌ Manual synchronization required

### Option B: docker-compose extends / merge files
- ⚠️ YAML merge is fragile with arrays
- ⚠️ Not natively supported in Compose v2 for all fields
- ❌ Still requires manual version management

### Option C: Terraform templatefile → single docker-compose.yml (CHOSEN)
- ✅ Single source of truth: all versions in `main.tf` `locals.versions`
- ✅ Terraform tracks file generation in state
- ✅ Template vars prevent version drift between services
- ✅ Idempotent: `terraform apply` idempotently regenerates `docker-compose.yml`
- ✅ Re-runnable: same vars → same output, always

---

## Decision

**All Docker Compose configuration is generated by Terraform from `docker-compose.tpl`.**

`docker-compose.yml` is a **generated artifact** — it must never be manually edited.

### Architecture

```
main.tf
├── locals.versions        ← ALL pinned service versions (single source of truth)
├── locals.network         ← ALL ports and network config
├── locals.storage         ← ALL volume paths (local + NAS)
├── locals.resources       ← ALL CPU/memory limits
└── docker_compose_vars    ← Assembles all vars for templatefile()
      │
      └── docker-compose.tpl  (templatefile())
            │
            └── docker-compose.yml  ← GENERATED — do not edit manually
```

### Adding a Service (Required Process)

1. Add version tag to `locals.versions` in `main.tf`
2. Add port(s) to `locals.network` in `main.tf`
3. Pass both to `docker_compose_vars` in `main.tf`
4. Add service block using template vars in `docker-compose.tpl`
5. Run `terraform apply` on host .31 to regenerate
6. Run `docker-compose up -d <service>` to start

### Monitoring Stack Integration

As of Phase 2 consolidation, the monitoring stack (prometheus, grafana, alertmanager, postgres, redis, jaeger) is fully integrated into `docker-compose.tpl` with:
- Pinned versions in `locals.versions`
- Port mappings in `locals.network`
- Internal Docker network communication (container names, not `localhost`)
- Proper health checks for all services

### Version Pin Enforcement

All versions live exclusively in `locals.versions`:
```hcl
locals {
  versions = {
    prometheus   = "v2.48.0"
    grafana      = "10.2.3"
    alertmanager = "v0.26.0"
    postgres     = "15-alpine"
    redis        = "7-alpine"
    jaeger       = "1.50"
    # ... all other services
  }
}
```

The template enforces: `image: prom/prometheus:${prometheus_version}` — the version cannot be set other than via Terraform.

---

## Consequences

### Positive
- Zero configuration drift: deployed state always matches IaC
- Version bumps happen in exactly one place (`main.tf`)
- `terraform apply` is safe to re-run — idempotent by design
- Generated `docker-compose.yml` is readable for debugging
- All phase-specific compose files archived/deleted

### Negative
- New service addition requires Terraform change (slight overhead)
- `docker-compose.yml` must not be edited directly (enforced by comment header)

### Mitigation
- `docker-compose.yml` contains warning: `# NOTE: This file is GENERATED by Terraform — DO NOT EDIT MANUALLY`
- CI validate step checks that `docker-compose.yml` matches `terraform apply` output
